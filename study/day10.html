요구사항 명세 — 제대로 적는 법

2026-07-13 · 오늘 배울 개념 4개 + 퀴즈 10문제 (약 1시간)

복습 한 스푼 🥄

어제의 핵심은 DFD 4요소 — 프로세스(원), 자료 흐름(화살표), 자료 저장소(평행선), 단말(사각형)이었어요. 자료 사전 기호 { }=반복, [ ]=선택, ( )=생략도요! 오늘은 3단계 명세 — 분석한 요구사항을 문서로 제대로 적는 방법입니다.

1. 요구사항 명세란?

명세(明細)는 '분명하고 자세하다'는 뜻이에요. 분석까지 끝난 요구사항을 모두가 같은 뜻으로 이해할 수 있게 문서로 적는 것이 명세 단계예요. 이 문서를 요구사항 명세서(SRS)라고 불러요.

비유 — 부동산 계약서예요. "적당한 가격에 좋은 집" 같은 애매한 말 대신 "보증금 5,000만 원, 월세 50만 원, 2년 계약"처럼 누가 읽어도 똑같이 이해되게 적죠.

2. 좋은 명세서의 조건

조건뜻
명확성한 가지 뜻으로만 읽혀야 함 (애매한 표현 금지)
완전성필요한 내용이 빠짐없이 있어야 함
일관성문서 안에서 서로 모순이 없어야 함
검증 가능성지켜졌는지 확인(테스트)할 수 있어야 함
추적 가능성각 요구가 어디서 왔고 어디에 반영됐는지 따라갈 수 있어야 함
나쁜 예 → 좋은 예 — "빨라야 한다"(검증 불가) → "검색 결과를 2초 이내에 표시한다"(검증 가능!). 숫자로 적으면 대부분 좋아져요.
시험 포인트 — "요구사항은 검증 가능해야 한다"가 핵심. '빠르게', '편리하게' 같은 애매한 표현은 나쁜 명세의 예로 출제돼요.

3. 정형 명세 vs 비정형 명세 ⭐

명세서를 적는 방법은 크게 둘로 나뉘어요.

비정형 명세정형 명세
표현 방법자연어(일상 언어) + 그림수학·논리 기호
장점누구나 쉽게 이해모호함이 없음, 기계로 검증 가능
단점사람마다 다르게 해석될 수 있음어려워서 배운 사람만 씀
대표 기법자연어 서술, FSM, 의사결정표Z(제트), VDM, 페트리넷
비유 — 비정형은 한글 편지, 정형은 수학 공식이에요. 편지는 읽기 쉽지만 오해의 여지가 있고, 공식은 어렵지만 오해가 없죠.
시험 포인트 — "Z, VDM, 페트리넷 = 정형 명세 기법"이 단골! 수학 기호가 나오면 무조건 정형이에요.

4. 요구사항 명세서(SRS)에 들어가는 것

SRS(Software Requirements Specification)에는 보통 이런 내용이 들어가요: 시스템의 목적과 범위, 기능적 요구사항(Day 6!), 비기능적 요구사항(성능·보안), 제약 조건, 용어 정의.

시험 포인트 — SRS는 개발자·고객·테스터 모두가 보는 공식 기준 문서예요. 이후 설계·테스트가 전부 이 문서를 기준으로 진행됩니다.
