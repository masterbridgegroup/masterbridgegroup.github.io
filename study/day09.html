Day 9 · 1과목 소프트웨어 설계

요구사항 분석과 DFD(자료 흐름도)

2026-07-12 · 오늘 배울 개념 4개 + 퀴즈 10문제 (약 1시간)

복습 한 스푼 🥄

어제는 요구사항 도출 기법들을 배웠죠 — 인터뷰, 브레인스토밍(비판 금지·질보다 양), 관찰, 프로토타입. 그리고 시스템 바깥의 사용자를 액터라고 불렀어요. 오늘은 2단계 분석 — 모아온 요구사항을 따져보고 그림으로 정리하는 방법입니다.

1. 요구사항 분석이란?

도출로 모은 요구사항들은 뒤죽박죽이에요. 분석 단계에서는 이걸 정리하면서 서로 모순되는 건 없는지, 빠진 건 없는지, 실제로 만들 수 있는지(타당성)를 따져요.

비유 — 가족 여행 계획을 짤 때 아빠는 "산", 엄마는 "바다", 아이는 "놀이공원"이라고 하면? 이 충돌을 조율하고 예산 안에서 가능한지 따지는 게 분석이에요.
시험 포인트 — 분석 단계의 키워드: 충돌(상충) 해결, 우선순위 결정, 타당성 검토.

2. 구조적 분석 도구 — DFD (자료 흐름도) ⭐

DFD(Data Flow Diagram)는 데이터가 시스템 안에서 어떻게 흘러다니는지를 그린 그림이에요. 구성 요소는 딱 4가지!

구성 요소그리는 모양뜻
프로세스원(동그라미)데이터를 가공·처리하는 작업
자료 흐름화살표데이터가 이동하는 길
자료 저장소평행선 2줄데이터가 보관되는 곳
단말사각형시스템 밖의 데이터 출발지/도착지 (사람·외부 조직)
외우는 법 — "프원, 흐화, 저평, 단사" 또는 이렇게: 일하는 곳은 동그란 공장(프로세스=원), 길은 화살표, 창고는 선반 두 칸(평행선), 바깥 손님은 네모 명함(사각형).
시험 포인트 — "프로세스는 사각형으로 표시한다" 같은 모양 바꿔치기 문제가 초단골! 4가지 요소·모양 짝을 정확히 외우세요.

3. 자료 사전 (DD)

자료 사전(Data Dictionary)은 DFD에 나오는 데이터가 정확히 무엇인지 기호로 정의한 사전이에요.

기호뜻예시
=정의 (~로 구성된다)주소 = 시 + 구 + 동
+연결 (그리고)이름 + 전화번호
( )생략 가능(회사명) — 있어도 없어도 됨
[ | ]선택 (둘 중 하나)[남 | 여]
{ }반복{주문상품} — 여러 개 가능
* *주석 (설명)*고객 식별용*
시험 포인트 — 기호 6개의 뜻을 묻는 문제가 매년 나와요. 특히 { }=반복, [ ]=선택, ( )=생략을 헷갈리지 마세요!

4. 요구사항 분석에 쓰는 CASE 도구

CASE(케이스) 도구는 소프트웨어 개발 과정을 자동으로 도와주는 프로그램이에요. 분석 단계에서는 요구사항을 추적하고, 일관성(앞뒤가 맞는지)을 자동으로 검사해줘요. SADT, PSL/PSA 같은 이름이 시험에 등장해요.

비유 — 맞춤법 검사기가 글의 오류를 자동으로 찾아주듯, CASE 도구는 요구사항 문서의 모순을 자동으로 찾아줘요.
