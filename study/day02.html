스크럼과 XP 자세히 보기

2026-07-05 · 오늘 배울 개념 4개 + 퀴즈 10문제 (약 1시간)

복습 한 스푼 🥄

어제 배운 애자일은 짧은 주기로 조금씩 만들어 자주 보여주는 방식이었죠. 애자일이 더 중요하게 여기는 4가지는 개인과 상호작용, 작동하는 소프트웨어, 고객과의 협력, 변화에 대응이었어요. 오늘은 애자일의 두 대표선수인 스크럼과 XP를 깊게 들여다봅니다.

1. 스크럼의 세 가지 역할

스크럼 팀에는 딱 3가지 역할이 있어요.

역할하는 일
제품 책임자(PO)"무엇을 만들지" 결정. 요구사항 목록(백로그)의 우선순위를 정함
스크럼 마스터(SM)팀이 일 잘하게 돕는 도우미. 방해 요소를 치워줌 (팀장·지시자가 아님!)
개발팀실제로 만드는 사람들 (보통 7~9명)
비유 — 식당으로 치면 PO는 메뉴를 정하는 사장, 스크럼 마스터는 주방이 원활히 돌아가게 돕는 지배인, 개발팀은 요리사들이에요.
시험 포인트 — "스크럼 마스터는 팀을 통제하는 관리자다" = 틀린 설명! 돕는 사람이지 지시하는 사람이 아니에요.

2. 스크럼이 굴러가는 순서

제품 백로그(할 일 전체 목록) → 스프린트 계획 회의 → 스프린트(1~4주 개발) → 매일 15분 일일 스크럼 회의 → 스프린트 검토 회의(결과물 시연) → 스프린트 회고(잘한 점·개선점 돌아보기)

용어 풀이 — 백로그(backlog)는 '밀린 일 목록', 스프린트(sprint)는 '전력 질주'라는 뜻이에요. 짧게 전력 질주하고, 돌아보고, 또 질주하는 거죠.

진행 상황은 번다운 차트(남은 작업량이 시간에 따라 줄어드는 그래프)로 확인해요.

시험 포인트 — 순서 문제 단골: 계획 → 스프린트 → 일일 회의 → 검토 → 회고. 그리고 "남은 작업량 그래프 = 번다운 차트".

3. XP의 실천 기법

XP(익스트림 프로그래밍)는 어제 배운 5가지 가치(용기·단순성·의사소통·피드백·존중) 위에 구체적인 실천 방법들을 얹어요. 시험에 자주 나오는 것만 추리면:

기법뜻
짝 프로그래밍두 명이 한 컴퓨터로 함께 코딩 (한 명은 작성, 한 명은 검토)
테스트 주도 개발(TDD)코드를 만들기 전에 테스트부터 먼저 작성
지속적 통합(CI)여러 사람이 만든 코드를 자주 합쳐서 문제를 일찍 발견
리팩토링기능은 그대로 두고 코드 내부만 깔끔하게 정리
소규모 릴리즈조금씩 자주 배포(공개)
공동 코드 소유코드는 팀 전체의 것 — 누구나 고칠 수 있음
시험 포인트 — "리팩토링은 기능을 바꾸는 것이다" = 틀림! 기능은 그대로, 내부 구조만 개선하는 거예요.

4. 스크럼 vs XP 한눈 비교

스크럼XP
초점일하는 방식·관리 (회의, 역할, 주기)개발 기술 자체 (코딩·테스트 방법)
주기 이름스프린트 (1~4주)릴리즈를 짧게 반복
키워드백로그, 스프린트, 회고짝 프로그래밍, TDD, 리팩토링
구분 요령 — 회의·역할 이야기가 나오면 스크럼, 코딩 방법 이야기가 나오면 XP라고 기억하세요.
